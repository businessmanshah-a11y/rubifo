# دستورالعمل جامع پیش‌تولید LuxCounter

**نسخه:** 2.0  
**تاریخ تدوین نسخه جدید:** 1405/02/18 | 2026/05/08  
**وضعیت:** مرجع اجرایی پیش‌تولید V1  
**مخاطب:** مالک محصول، CTO، تیم طراحی، تیم توسعه، DevOps، دستیارهای AI  
**مبنای اولیه:** `docs/سند پروسه پیش تولید.pdf`  
**نسخه قابل ویرایش:** این فایل Markdown منبع به‌روز و قابل توسعه دستورالعمل پیش‌تولید است.

---

## 1. هدف سند

هدف پیش‌تولید این است که قبل از نوشتن کد production، تمام تصمیم‌های محصول، طراحی، معماری، اجرا، ابزارها، دامنه پروژه، ریسک‌ها و مسیر توسعه روشن شوند.

در پایان پیش‌تولید باید بتوانیم بگوییم:

```text
هر انسان یا AI با ورود به این repository می‌داند محصول چیست، ظاهر آن دقیقاً بر اساس چه مرجعی است، با چه معماری ساخته می‌شود، چه taskهایی دارد، هر task چگونه گزارش می‌شود، و چه چیزهایی خارج از V1 است.
```

---

## 2. اصول پیش‌تولید

1. **هیچ کدنویسی production قبل از شفاف شدن سندها انجام نمی‌شود.**
2. **Git و docs منبع حقیقت پروژه هستند.**
3. **هر تصمیم مهم باید در سند مربوطه ثبت شود.**
4. **پروتوتایپ بصری می‌تواند با ابزار AI ساخته شود، اما معماری production مستقل و کنترل‌شده باقی می‌ماند.**
5. **AIها فقط طبق WBS، tracker و instructionها کار می‌کنند.**
6. **هر task باید وضعیت، خروجی، verification، bug، risk و decision خود را ثبت کند.**
7. **هر چیزی که در V1 نیست، باید صریحاً خارج از محدوده یا post-V1 ثبت شود.**

---

## 3. نمای کلی مسیر پیش‌تولید

مسیر نهایی پیش‌تولید LuxCounter در 10 فاز انجام می‌شود:

| فاز | عنوان | خروجی اصلی |

| P0 | آماده‌سازی SSOT | فولدر docs و سند پروسه |

| P2 | تفکیک فیچرها و ماژول‌ها | Feature Master Plan |

## 4. P0 — آماده‌سازی SSOT

### خروجی LuxCounter


### Definition of Done

- همه اعضا و AIها می‌دانند تصمیم‌ها باید در docs ثبت شوند.

## 5. P1 — تعریف بیزینس و محصول

### هدف

شفاف‌سازی چرایی، مخاطب، مدل درآمدی، KPIها و محدوده محصول.

### سوال‌های کلیدی

- محصول دقیقاً چه مشکلی را حل می‌کند؟
- مخاطب B2C کیست؟
- مخاطب B2B کیست؟
- هدف اصلی conversion چیست؟
- موفقیت با چه KPIهایی سنجیده می‌شود؟
- چه چیزهایی خارج از فاز اول هستند؟

### خروجی LuxCounter

```text
docs/__سند نیازمندی_های محصول (Product Requirements Document - PRD)__.md
```

### تصمیم‌های کلیدی ثبت‌شده

- LuxCounter ماشین تولید لید و اعتمادسازی است، نه فروشگاه پرداخت آنلاین.
- مسیرهای B2C و B2B باید جدا و قابل اندازه‌گیری باشند.
- قیمت به‌روز و استعلام سریع مزیت اصلی محصول است.

### Definition of Done

- PRD کامل و قابل ارجاع باشد.
- اهداف V1 و Non-Goals روشن باشند.

## 7. P3 — تصمیم معماری فنی

### هدف

انتخاب معماری، فناوری‌ها، زیرساخت، مرز بک‌اند/فرانت‌اند و مسیر رشد آینده.

### خروجی LuxCounter

```text
docs/_سند جامع معماری فنی و دستورالعمل توسعه.md
```

### تصمیم‌های کلیدی

- Frontend: Next.js
- Backend/Core: Headless WordPress
- API: REST first
- Logic اختصاصی: WordPress plugin با نام `luxcounter-core`
- DevOps: Next.js روی PaaS پارس‌پک، WordPress روی هاست وردپرسی پارس‌پک
- GraphQL، AI، 3D، CRM کامل و موتورهای سنگین به آینده موکول می‌شوند.

### Definition of Done

- ساختار فولدرها روشن باشد.
- API strategy روشن باشد.
- ساختار plugin روشن باشد.
- DevOps اولیه روشن باشد.

## 8. P4 — مهندسی طراحی و مرجع بصری

### هدف

رسیدن به ظاهر نهایی، قفل کردن design tokens و تعیین مرجع بصری غیرقابل‌انحراف.

### مسیر طی‌شده در LuxCounter

1. بررسی پروتوتایپ اولیه `Luxury-Landing 2`
2. تولید پروتوتایپ جدید با AI/Lovable
3. جایگزینی مرجع بصری با خروجی بهتر در `docs/lovable`
4. ثبت قانون controlled port
5. ثبت فونت Peyda به عنوان فونت فارسی production

### خروجی‌ها

```text
docs/_سند سیستم طراحی و توکن‌های بصری.md
docs/lovable
docs/Peyda
```

### قانون طلایی طراحی

```text
Production UI باید از نظر بصری با docs/lovable یکی باشد.
```

اما:

```text
کل اپ Lovable نباید پایه production شود؛ فقط کامپوننت‌ها، tokens، interactionها و data shape به Next.js port می‌شوند.
```

### Definition of Done

- مرجع بصری نهایی مشخص باشد.
- فونت، رنگ، theme، زبان و responsive rules مشخص باشند.
- اجزای قابل port و غیرقابل port مشخص باشند.

## 12. P8 — سیستم عامل اجرای AI

### هدف

تضمین اینکه Codex، Claude، Cursor یا هر AI دیگر طبق سندها کار کند و گزارش دهد.

### خروجی‌ها

```text
AGENTS.md
CLAUDE.md
.cursor/rules
docs/_ردیاب اجرا و وضعیت پروژه.md
```

### قوانین اصلی

- هر AI باید task ID از WBS انتخاب کند.
- قبل از کار، tracker را به `In Progress` تغییر دهد.
- بعد از کار، tracker را به `Done`، `Blocked` یا `Deferred` تغییر دهد.
- bugs، risks و decisions را ثبت کند.
- خارج از scope کار نکند.
- visual drift از `docs/lovable` ایجاد نکند.

### Definition of Done

- دستورالعمل Codex وجود داشته باشد.
- دستورالعمل Claude وجود داشته باشد.
- rules برای Cursor وجود داشته باشد.
- tracker وضعیت پروژه وجود داشته باشد.

## 16. قانون به‌روزرسانی آینده

اگر در آینده تصمیم جدیدی گرفته شد:

1. سند مرتبط باید به‌روزرسانی شود.
2. اگر روی scope اثر دارد، WBS باید به‌روزرسانی شود.
3. اگر روی اجرا اثر دارد، tracker باید decision log بگیرد.
4. اگر روی AIها اثر دارد، `AGENTS.md`، `CLAUDE.md` یا `.cursor/rules` باید اصلاح شود.
5. اگر روی deploy یا content اثر دارد، چک‌لیست مربوطه باید اصلاح شود.

هیچ تصمیم شفاهی نباید بدون ثبت در docs مبنای اجرا قرار بگیرد.
